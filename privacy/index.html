1. Overview

This Privacy Policy explains what data Vigil ("the Bot", "we", "us") collects, how it is stored, who can access it, how long it is kept, and your rights regarding it. By adding or using Vigil, you agree to this policy and the Terms of Service.

2. Data We Collect

Vigil collects and stores only the data required to operate its security and moderation features:

  • Guild IDs ‚Äî to scope all stored data to the correct server
  • Channel IDs ‚Äî for log channel configuration and verification setup
  • Role IDs ‚Äî for owner, trusted admin, whitelist, and verification role assignments
  • User IDs ‚Äî for whitelist, quarantine, and permission records
  • Quarantine records ‚Äî reason, timestamp, and saved role list per quarantined user
  • Configuration settings ‚Äî thresholds, permission tiers, and feature toggles
  • Backup data ‚Äî server name, full role list, channel structure, emojis, and stickers
  • Error logs ‚Äî command names, timestamps, and guild identifiers for debugging purposes; user IDs are only included where directly relevant to the error

Vigil does not collect message content, direct messages, email addresses, passwords, IP addresses, or payment information.

3. Verification Codes

When a member clicks Verify, a temporary 4-digit code is generated and held in memory only ‚Äî never written to disk ‚Äî for a maximum of 5 minutes. It is deleted immediately after use or expiry.

4. How Data Is Used

  • Detecting and responding to raids, anti-nuke events, and other security threats
  • Performing automated and manual moderation actions
  • Maintaining and restoring server configuration backups
  • Providing security scoring and activity logging
  • Managing the three-tier permission hierarchy
  • Developer debugging, maintenance, and schema migration

Data is never used for advertising, profiling, or any purpose beyond operating the bot.

5. Who Has Access

Stored data is accessible to:

  • Server administrators ‚Äî via bot commands or direct access to the data/ and backups/ directories on the hosting server
  • The bot developer(s) ‚Äî solely for maintenance, debugging, security, or data migration purposes. This includes inspecting corrupted backups, diagnosing restore failures, and performing global backup operations

No other parties have access to stored data.

6. Automated Decision-Making

Vigil uses automated systems to detect suspicious behaviour and may automatically quarantine, ban, or restrict users. False positives can occur. Server administrators are responsible for reviewing and correcting automated actions.

7. Data Storage

All data is stored locally on the server hosting Vigil as plain-text JSON files. Data is not encrypted by default. The bot operator is responsible for securing the hosting environment, restricting file access, and applying encryption at rest if required.

Vigil uses atomic file writes to prevent corruption, sanitizes backup names to prevent path traversal, and auto-recovers from corrupted files where possible.

8. Data Retention

  • Whitelist / Owner / TA records ‚Äî until removed via bot commands
  • Quarantine records ‚Äî until the user is released
  • Permanent backup slots (1‚Äì5) ‚Äî until overwritten by a new backup in that slot
  • Auto backup slots (AB1‚ÄìAB3) ‚Äî rolling 3-day history; the slot being written to overwrites its previous file each cycle
  • Verification codes ‚Äî 5 minutes maximum, held in memory only
  • Error logs ‚Äî retained at the discretion of the bot operator; not stored in a separate log file by default (printed to console only)

Removing Vigil from your server does not automatically delete stored data files. The bot operator is responsible for purging data for inactive servers. Data may also be deleted by the developer during maintenance, migration, or following prolonged inactivity.

9. Backup Operations by the Developer

The developer may trigger backup operations across all guilds for maintenance, security, or migration purposes. These operations follow the same round-robin slot system (AB1/AB2/AB3) as scheduled daily backups and are logged to each guild's configured log channel. No additional data is collected during these operations.
